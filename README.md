# Kingdom 4

Kingdom 4 is a small, self-contained “AI kingdom” designed to be easy to start with and flexible enough to grow over time. It is built around three main roles: a Commander, one or more Knights, and a Security Node. Everything lives in a single repository so that anyone can clone it once, answer a few simple questions, and let the system configure itself. You do not need to hard-code IP addresses or usernames into the code. Instead, the setup script asks you what this machine should be and how you want it to behave.

The Commander is the central control node. It runs a FastAPI application that exposes both a JSON API and a simple web-based UI. From the Commander’s web UI, you can see basic system information, a list of detected tools, and security alerts generated by the Security Node. The Commander also provides an endpoint for Knights to register themselves. When a Knight registers, it sends information about its hardware capabilities, and the Commander assigns it a level: Level 1, Level 2, or Level 3. These levels are used to decide how heavy or light the task engine should be on that Knight.

Knights are worker nodes. A Knight can be a laptop, desktop, virtual machine, or small device. Each Knight runs a lightweight Python agent that asks you for the Commander’s IP or hostname, scans its own hardware using a small capability module, and then sends that information to the Commander. The Knight does not need to know anything about the rest of the system. It just needs the repository, Python, and the ability to reach the Commander over the network. This makes it easy to add new Knights as you go.

Kingdom 4 includes three variants of a task engine inspired by the idea of an orchestration core: a full engine, a lite engine, and a micro engine. The full engine is intended for powerful machines (Level 1 Knights) that can handle more complex workflows. The lite engine is for mid-range machines (Level 2 Knights) that can do some orchestration but should not be overloaded. The micro engine is for low-power devices (Level 3 Knights) that should only run simple, single-step tasks. The Commander decides which engine a Knight should use based on its assigned level, but Knights can also request a specific level if you want to override the automatic choice.

A Security Node component runs on the Commander side. It is not a separate machine in this starter version, but a logical role inside the Commander process. The Security Node periodically checks recent system logs and running processes for simple patterns, such as the presence of “error” in logs or suspicious-looking process names. When it finds something, it records an alert that is exposed through the Commander’s status endpoint and visible in the web UI. This is not meant to be a full security solution, but it gives you a starting point for thinking about monitoring and safety in your own small network.

The repository also includes a tool scanning module. This module looks for a few common tools installed on the machine, such as Python, Docker, Git, and others. In a more advanced version, this could be extended to look up tools online and ask the user for clarification when something unknown is found. In this initial version, the scanner focuses on detecting a small set of tools and making that information available to the Commander. This helps you understand what capabilities each node has without manually checking each machine.

One of the main goals of Kingdom 4 is to be friendly to people who are new to GitHub and distributed systems. The repository includes a setup script that asks you simple questions: whether this machine should act as a Commander, a Knight, or both; which port to use for the web UI; and how you want to run the services. After you answer these questions, the script creates a Python virtual environment, installs the required dependencies, and prints clear next steps. You can then start the Commander, start a Knight, or do both on the same machine.

The web UI is served from the same FastAPI application as the Commander API. Once the Commander is running, you can open a browser and go to the configured port on that machine to see a simple dashboard. With basic network configuration, such as port forwarding on your router or using a tunneling service, you can access this UI from other devices that are not on your local network. This allows you to treat your Commander as a local “AI hub” that you can reach from your phone, tablet, or another computer, without hard-coding any network details into the code.

Kingdom 4 is intentionally minimal in its first version. It does not assume any specific cloud provider, identity system, or external database. Everything runs locally using Python, FastAPI, and a few standard libraries. The repository is structured so that you can extend it over time: adding more modules, more Knight types, more security checks, or more advanced orchestration logic. The goal is to give you a clear, understandable foundation that you can grow into your own personal or shared AI kingdom, starting from a single GitHub clone and a few simple questions.
